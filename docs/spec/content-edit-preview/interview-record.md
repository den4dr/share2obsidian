# 展開内容の編集・プレビュー機能 ヒアリング記録

**作成日**: 2026-03-29
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

既存の share-content-expansion 実装（14タスク完了済み）を基盤として、編集・プレビュー機能に必要な追加要件を明確化するためのヒアリングを実施しました。

## 質問と回答

### Q1: 編集・プレビュー画面の表示タイミング

**質問日時**: 2026-03-29
**カテゴリ**: 未定義部分詳細化
**背景**: 全コンテンツタイプで表示するか、一部のみ（URL等）に絞るかで実装コストが大きく変わるため

**回答**: 常に表示（全コンテンツタイプで処理後に必ず編集画面を表示）

**信頼性への影響**:
- REQ-001（全コンテンツタイプで編集画面表示）の信頼性が 🔴 → 🔵 に向上

---

### Q2: 編集できる項目

**質問日時**: 2026-03-29
**カテゴリ**: 未定義部分詳細化
**背景**: どの項目まで編集可能にするかによってUIの複雑度と実装コストが変わるため

**回答**: タイトル（title）・本文（body）・タグ（tags）・保存先フォルダの4項目

**信頼性への影響**:
- REQ-003（編集フィールド4項目）、REQ-405（フォルダ初期値）の信頼性が 🔴 → 🔵 に向上

---

### Q3: プレビュー機能の要否

**質問日時**: 2026-03-29
**カテゴリ**: 既存設計確認
**背景**: Markdown プレビューはライブラリ（Markwon 等）が必要で工数増になるため、要否を確認

**回答**: プレビューなし（編集画面のみ）

**信頼性への影響**:
- プレビュー機能の要件を除外。実装スコープを編集画面に絞り込み

---

### Q4: URL 展開中のローディング画面の扱い

**質問日時**: 2026-03-29
**カテゴリ**: 影響範囲
**背景**: 既存の `LoadingScreen` を URL 処理中に使用しており、編集画面追加後の遷移フローを確認するため

**回答**: ローディング後に編集画面（URL展開完了後に編集画面へ遷移）

**信頼性への影響**:
- REQ-301（ローディング後に編集画面へ遷移）の信頼性が 🔴 → 🔵 に向上

---

### Q5: アクションボタンの構成

**質問日時**: 2026-03-29
**カテゴリ**: 未定義部分詳細化
**背景**: 送信・キャンセル以外にクリップボードコピー等のボタンを追加するかを確認

**回答**: 送信・キャンセルの2ボタンのみ

**信頼性への影響**:
- REQ-004（送信・キャンセルボタン）の信頼性が 🔴 → 🔵 に向上

---

### Q6: タグ入力形式

**質問日時**: 2026-03-29
**カテゴリ**: 未定義部分詳細化
**背景**: Chip UI vs テキスト入力でUIの実装コストが異なるため

**回答**: カンマ区切りテキスト（例: `shared, web, clipping`）

**信頼性への影響**:
- REQ-103（カンマ区切りパース）の信頼性が 🔴 → 🔵 に向上

---

## ヒアリング結果サマリー

### 確認できた事項
- 全コンテンツタイプで編集画面を挟む（シンプルな統一フロー）
- 編集項目は4項目（タイトル・本文・タグ・フォルダ）
- プレビュー機能はスコープ外（実装を簡素化）
- 既存 LoadingScreen はそのまま活用可能
- ボタン構成はシンプルな2ボタン

### 追加/変更要件
- 追加: 編集画面 Composable の新規実装
- 追加: ViewModel によるフォーム状態管理
- 追加: カンマ区切りタグ文字列 → `List<String>` 変換ロジック
- 変更: `MainActivity` のフロー（即 Obsidian 起動 → 編集画面経由に変更）

### 残課題
- ViewModel を使うか Compose `remember` で状態管理するか（実装時判断）
- 本文フィールドの最大行数/スクロール動作の詳細
